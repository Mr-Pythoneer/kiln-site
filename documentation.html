§00

Overview

p.1

Kiln is a done-for-you service that builds one private, specialist model for one job you do over and over. You bring examples of the task done right; we specialise an open base model on those examples and hand back a model that is yours. Not an account, not an API key with a meter running — a file you keep, run, and control.

It is for the narrow, repeated, sensitive work that a giant general chatbot handles well enough but that you would rather not ship off-site every time: classifying tickets, extracting fields from your own documents, rewriting into your house style, drafting the same kind of structured output a hundred times a day. A model that only does your task can be small, fast, and cheap to run — and because it runs on your hardware, the data never leaves.

What you walk away with is open weights delivered as a single GGUF file, sized to where you plan to run it, plus what you need to run it locally with Ollama or llama.cpp. No dependency on us afterward. If you never talk to Kiln again, the model keeps working exactly as it did the day we handed it over.

§01

Own it, don't rent it

p.2

The reflex, when you need a model to do something, is to reach for a general endpoint in the cloud. For open-ended work that reflex is right. For a narrow, repeated, sensitive job it is the wrong default — you are renting a vast general-purpose system to do one small thing, paying per call to do it, and sending your data to someone else's machine every single time to get it done.

A model that only has to do your task doesn't need to be vast. Specialised on your examples, a small model can match or beat the general endpoint on the one thing you care about, run on hardware you already own, and cost nothing per call because there is no call — it's just a program running locally. And it doesn't drift: the endpoint you rent can change under you next week, but a model you own is frozen at the behavior you tested and signed off on.

THE RENTED ENDPOINT

Someone else's model, metered
  • Billed per call, forever — cost scales with use
  • Your data leaves the building on every request
  • General-purpose, carrying weight you never use
  • Can change or be deprecated without your say
  • Needs a network and their servers to work at all

AN OWNED SPECIALIST

Your model, yours to run
  • Free to run — no meter, no per-call cost
  • Runs local and offline; data never leaves
  • Specialised on your task, small and fast
  • Frozen at the behavior you tested and approved
  • A file you keep, license, and control outright

§03

How it works

p.4

An engagement with Kiln is short and concrete. You come with a narrow task and examples of it done well; we specialise an open base model on those examples and hand back a model you own. There is no platform to log into, no monthly seat, and nothing of yours living on our machines after the work is done.

The whole thing runs in three passes: we agree on what we are building, we build it and measure it against your standard, and we deliver the weights for you to run yourself. Most first models go from first conversation to a file on your disk in a matter of days, not months. Here is each pass in order.

STEP 01 / SCOPE

We agree on the job

A short conversation about three things: the exact task you want automated, the kind of data you can show us, and the hardware it has to run on. This is also where we are honest with you. If a specialised small model is the wrong tool, or your task is too broad to pin down, we say so before you spend anything.

STEP 02 / BUILD

We specialise and grade

You send your examples. We specialise an open base model on them and test the result against held-out cases it has never seen, scored by your definition of correct rather than a generic benchmark. How we train is our business; what matters to you is that we keep tuning until it clears the bar you set.

STEP 03 / DELIVER

You receive and run it

You get the model weights as a single file plus a short, plain report on how it performed on those held-out cases. You install it and run it yourself, offline, on your own machines. If it needs to get sharper or cover new cases, we can run follow-up rounds.

scope the task → specialise on your examples → grade against your standard → deliver weights you own

Nothing here is one-and-done. Your task will drift, your edge cases will multiply, and a good model is one you can revisit. Follow-up rounds fold in new examples and correct the misses you find in real use, so the model tracks the job as the job changes. You are never locked in: because you hold the weights, you can keep running the version you have for as long as it serves you, with or without us.

§04

The data you provide

p.5

A good dataset is simple to describe: paired examples of your task done the way you would sign off on it. Each example is an input — the thing your model will see in the real world — and the output you want back for it. Nothing more exotic than that. If you can show us a stack of cases where the answer is already right, you have most of what we need.

The model learns the pattern in those pairs: your format, your judgement calls, your house style. That means the pairs have to reflect the standard you actually hold, not a rough approximation. The table below is what we look for.

QuestionGuidance
What a row looks likeOne input paired with the ideal output for it — the exact response, classification, or transformation you would want back. A support ticket and the reply you would approve; a contract clause and the summary you would file.
Accepted formatsJSONL, CSV, or a plain spreadsheet all work. If your examples live in an export from some other system, send it as-is and we help you shape it into pairs.
How muchA few dozen is enough to pilot and see whether the task is learnable. A few hundred to a few thousand gets you a dependable model. Past that, quality beats volume every time — clean examples move the model more than bulk.
What makes it goodReal cases from your actual work, outputs that are genuinely correct, coverage of the edge cases that trip you up, and a consistent style across rows so the model learns one voice instead of several.
What to avoidThin data that only shows the easy path, contradictory rows where similar inputs get different answers for no reason, and one-note sets that repeat a single scenario. Any of these teach the model the wrong lesson.
How you send itThrough your own secure link, on your terms. Your data is used only to build your model and is deleted afterward. It never trains anything but the model you are paying for.

You do not need a perfect, finished dataset to begin. Almost no one has one. Most engagements start from a messy export, a shared drive of past work, or a spreadsheet someone kept by hand — and part of what we do in the early rounds is help you turn what you already have into examples worth training on.

§05

Model size & where it runs

p.6

Size is a runtime decision, not a vanity one. A bigger model is not automatically a better one for your task, and a model that will not fit the machine you plan to run it on is worth nothing to you. So we work backwards from where the model will live. We start from an open base model sized to that target, specialise it on your examples, and stop at the smallest thing that clears your bar.

The win is a model that fits your hardware, runs offline, and gives you the answers you need — not the largest model we could hand you. When a small specialist matches a general giant on the one job you care about, the small specialist wins, because it is cheaper to run and yours to keep.

Runs onRough sizeGood for
A laptopSmallQuick, on-device, single-user tasks. No server, no network — the model runs next to your work.
A workstation or one GPUMidThe common sweet spot. Room for a strong specialist that still runs on hardware one person or one team already owns.
A single serverLargerHeavier reasoning and higher throughput — many requests, longer answers, or a tougher task that a mid model cannot hold.
An edge deviceCompactOffline and in-the-field. Small footprint for embedded and disconnected settings where every megabyte and watt counts.

Context length — how much text the model reads at once — is matched to your task, not maxed out for its own sake. A model that classifies short records needs far less than one that reasons over a long document, and a shorter window costs less to run. We size it to what your examples actually demand.

You can also receive smaller quantized builds of the same model — lighter files that trade a little quality for a smaller footprint, so you can run on lighter hardware or keep a fast fallback alongside the full-quality build. The size-versus-quality options are defined later in Delivery.

§06

Evaluation & quality

p.7

Before we specialise anything, we hold back a slice of your examples and keep it sealed — the model never sees it during training. That held-out slice is what we grade against afterwards, so the score reflects how the model does on work it has never encountered, not how well it memorised the examples it was shown.

Then we grade its answers against your definition of correct, because "right" means something different for every task. For code, does it run and produce the expected result. For extraction, does the pulled field match the true value. For a domain task, whatever "right" means in your world — the standard is yours, and we grade to it rather than to a generic benchmark.

HELD-OUT TEST
A slice of your examples is set aside and never trained on. The model is measured only on answers to inputs it has not seen before.
YOUR DEFINITION OF CORRECT
We grade to your standard — runs, matches, or otherwise passes — not to an off-the-shelf benchmark that has nothing to do with your task.
BEFORE & AFTER
We compare the specialised model to the untouched base model on the same held-out test, so the gain is measured and real, not asserted.
THE REPORT
A short, plain quality report ships with the model — what we tested, what we saw, in numbers you can read without a glossary.
ACCEPTANCE
You see the numbers before you commit. The result stands on the measurement, not on our say-so.
HONEST LIMITS
This is pre-beta. We do not promise a specific score in advance — we run the test and tell you what we actually saw.
§07

Delivery & running your model

p.8

What you get back is a model file, not access to a service. Kiln delivers your specialist as open weights in the GGUF format — the standard, portable container that the common local runtimes read directly. You download it once and it is yours: a single file you can copy, back up, and put on whatever machine will run it.

You run it with the tools you already know. Point Ollama at the file, or load it with llama.cpp or any compatible runtime, and it answers. There is no account to create, no key to paste, and no internet connection required — once the file is on your disk, the model runs entirely on your own hardware.

The same model is delivered in several builds. These are quantization levels — different trade-offs between file size and fidelity of the same specialist. Heavier builds reproduce the trained model more exactly; lighter builds shrink the file and the memory it needs to run, at a small and usually invisible cost. Pick by the hardware you will run on.

§08

Ownership & privacy

p.9

Ownership here is literal. When Kiln hands back your specialist, you hold the weights — the actual model file — with no seat count, no license server, and no runtime that reports to us. You can keep it, copy it across machines, deploy it to a fleet, or archive it for a decade. Nothing you run afterward touches Kiln, and nothing we operate is in the path of your model answering a prompt.

Privacy follows from where the model runs, not from a promise. Your specialist executes on your hardware, next to the data it works on, so sensitive inputs never leave your environment to be processed. There is no API to intercept and no telemetry to leak, because there is no outbound connection at all. Privacy is structural — a property of the architecture, not a policy you have to trust.

§09

Where it fits

p.10

A specialist model is not a general assistant. It is the right choice when the work is narrow, repeated, and valuable — one shape of task you run thousands of times, where getting it right matters and the format is predictable. That is exactly the ground where a small model, trained on your examples, can match or beat a much larger general model while running on hardware you already own.

If your need is broad, open-ended conversation across every topic, a frontier general model is the better tool. If it is a specific job done the same way every time, read on.

Use caseThe taskWhy a specialist wins
Code & DSL generationTurn plain requests into code, config, or a niche domain language your team uses.It learns your dialect and conventions instead of guessing at a generic average.
Structured extractionPull clean fields — dates, amounts, entities — out of messy documents into strict JSON.Trained on your schema, it holds the output shape far more reliably than a prompt.
A domain assistantAnswer questions and draft text inside one field, in your house style and vocabulary.It speaks your domain natively rather than hedging like a generalist.
Classification & taggingSort tickets, messages, or records into your categories at high volume.A small tuned model is fast and cheap enough to run on every item, all day.
Regulated / on-premHandle sensitive data where nothing may leave your network or reach a third party.You own the weights and run them in-house, so the data never crosses the wire.
Edge & offlineRun useful inference on a laptop, a device, or an air-gapped box with no cloud.A right-sized model fits the hardware and keeps working with the network unplugged.

§12

Glossary

p.13

Plain-language definitions for the terms used throughout this datasheet. No jargon assumed.

Fine-tuning
Taking a model that already understands language in general and specialising it on examples of one specific task, so it does that task well. This is the core of what Kiln does with your examples.
Base model
The open, general-purpose model we start from before specialising it. It knows a lot about language broadly but nothing about your particular task yet.
Specialist vs. general model
A general model tries to do everything passably; a specialist model does one job reliably. Kiln delivers a specialist — smaller and sharper on your task than a general model many times its size.
Weights
The numbers inside a model that hold everything it has learned. Running the model means doing arithmetic with these numbers; owning the weights means owning the model itself.
Open weights
Weights you receive as a file and are free to keep, copy, and run yourself, with no licence server and no dependence on the maker. Your delivered model is open weights — yours outright.
GGUF
The standard single-file format we deliver the model in. It packs the whole model into one portable file that common local tools such as Ollama or llama.cpp can run directly.
Quantization
Storing the model's numbers at lower precision to shrink the file and speed it up, at a small cost to quality. Levels like Q4_K_M through Q8_0 let you trade size against fidelity for the machine you run on.
Context length
How much text the model can consider at once — your input plus its output. A longer context lets it handle bigger documents in one pass but uses more memory.
Local / offline inference
Running the model on your own hardware with no internet connection. Your inputs never leave the machine, and the model answers without contacting any outside service.
Held-out test set
A portion of your examples deliberately kept aside so the model never sees them while it is built. Measuring against them shows how it performs on work it has genuinely not encountered.
Ownership
You hold the weights and run them on your own terms, indefinitely, with no seat limits, no subscription, and no way for us to switch them off. The model is yours to keep.
